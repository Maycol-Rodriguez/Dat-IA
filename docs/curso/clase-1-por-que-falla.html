Por qué falla un Text-to-SQL sin validación

"El SQL ejecutó sin error" es la barra más baja posible para confiar en una respuesta. Esta clase separa los 4 niveles de error que puede tener un SQL generado por un LLM, y por qué el pipeline actual de Dat-IA solo detecta el más barato de los cuatro.

1. El problema

Un usuario de negocio pregunta: "¿Cuál fue el ingreso promedio por pedido en Sao Paulo en 2018?". El optimizer de Dat-IA (app/optimizer/query_optimizer.py) detecta correctamente intent="aggregation", metrics=["revenue"], un filtro state=SP y date_range 2018-01-01..2018-12-31.

El generador (build_rag_response, app/main.py:889) recibe el DDL de olist_orders_dataset y olist_order_items_dataset y produce este SQL:

SQL generado (ejecuta sin error)
SELECT SUM(oi.price) AS total_revenue
FROM olist_orders_dataset o
JOIN olist_order_items_dataset oi ON o.order_id = oi.order_id
JOIN olist_customers_dataset c ON o.customer_id = c.customer_id
WHERE c.customer_state = 'SP'
  AND o.order_purchase_timestamp BETWEEN '2018-01-01' AND '2018-12-31';

Corre. Devuelve un número. execute_sql (app/main.py:937) no lanza error. Hoy eso basta para que /query/answer marque execution_status="success", validated=True (app/main.py:1370-1377) y lo guarde en memoria como ejemplo autoritativo.

🐛

Pero la pregunta pidió el promedio (AVG) y el SQL calcula la suma (SUM). Además el JOIN con olist_order_items_dataset multiplica cada pedido por su número de ítems: si un pedido tiene 3 productos, su price se cuenta 3 veces en total, no una. El número es real, ejecuta perfecto, y está mal en dos formas distintas.

2. El concepto desde cero: los 4 niveles de error

Cuando un LLM traduce lenguaje natural a SQL, el error puede vivir en capas muy distintas. Ordenarlas de barata a cara de detectar es la base de todo lo que viene en las clases siguientes.

NivelQué fallaEjemploCómo se detecta
1. Sintáctico El SQL no es SQL válido SELECT FROM WHERE mal formado, paréntesis sin cerrar Parsear el texto (gratis, sin BD)
2. De esquema Sintaxis válida, pero cita tablas/columnas que no existen customer_state en vez de state Comparar contra el DDL o un dry-run
3. Semántico Ejecuta, pero el resultado numérico está distorsionado por la estructura de la query El JOIN duplica filas; SUM en vez de AVG Requiere entender la intención, no solo el esquema
4. De negocio Ejecuta y el número es correcto, pero contesta otra pregunta Pidieron 2018, el filtro de fecha quedó en 2017 Requiere comparar contra la intención estructurada del usuario

El nivel 1 y buena parte del 2 se resuelven sin ningún LLM — son determinísticos y casi gratis (Clase 2). Los niveles 3 y 4 necesitan juicio, y ahí es donde entra un segundo LLM actuando como juez (Clase 3).

Principio de capas: barato antes que caro

La regla de oro de cualquier sistema de validación con LLMs: filtra con lo determinístico primero. Un parser de SQL cuesta microsegundos y cero tokens. Un juez LLM cuesta una llamada a la API completa. No tiene sentido gastar esa llamada en un SQL que ni siquiera parsea.

Costo relativo por capa (orden de magnitud)

Parsear sintaxis (sqlglot)~1 ms, $0
Verificar tablas contra DDL recuperado~1 ms, $0
Dry-run EXPLAIN en Postgres~10–50 ms, $0
Juez LLM (Gemini, salida estructurada)~500–1500 ms, costo de 1 llamada
Ejecutar SQL real + redactar respuesta~1–3 s, costo de 1–2 llamadas

3. Cómo se ve en Dat-IA hoy

El flujo actual de /query/answer (app/main.py:1264-1387) no tiene ningún nodo entre generación y ejecución:

app/main.py — flujo actual (simplificado)
rag_response = build_rag_response(query_for_generation, resp.ddl, memory_examples=memory_examples)
# nada entre estas dos líneas hoy
execution = execute_sql(sql_database, rag_response.sql)
# si no hay "error" en execution, se asume éxito y se marca validated=True

La causa raíz es estructural: build_rag_response devuelve un RAGResponse con status y confidence_note, pero esos campos los rellena el mismo LLM que generó el SQL — no hay una segunda opinión independiente. Un LLM casi nunca reporta baja confianza en su propio error.

4. Errores comunes al razonar sobre esto

  • Confundir "ejecuta" con "es correcto". Postgres no sabe qué pregunta de negocio intentabas responder; solo sabe si la sintaxis y los tipos son válidos.
  • Pensar que más ejemplos en el prompt resuelven el nivel 3–4. Ayudan, pero no eliminan la necesidad de verificar la salida — mitigan la tasa de error, no la anulan.
  • Validar todo con un solo LLM. Si el mismo modelo que generó el SQL también lo valida, hereda el mismo punto ciego (ver sesgo de self-preference, Clase 3).

5. Checklist de lo que viene

  • Nivel 1–2 (sintaxis + esquema): Clase 2, con sqlglot y EXPLAIN.
  • Nivel 3–4 (semántico + negocio): Clase 3, con un juez LLM que verifica contra la estructura del OptimizedQuery.
  • Qué pasa cuando algo falla: Clase 4, bucle de reintento con límite.
  • Qué pasa después de ejecutar: Clase 5, guardrail sobre el resultado y la respuesta redactada.

Comprueba que entendiste

Porque un motor SQL solo valida sintaxis, tipos y existencia de columnas/tablas — no sabe cuál era la intención de negocio original. Un JOIN que duplica filas o una métrica equivocada ejecutan perfectamente y devuelven un número, solo que el número está mal.
Porque son capas casi gratis (milisegundos, sin costo de API) que capturan los errores más comunes y baratos de detectar. Gastar una llamada a un LLM juez en un SQL que ni siquiera parsea es desperdiciar latencia y costo en un caso que un parser resuelve solo.
Nivel 3, semántico: el SQL es sintácticamente válido y usa tablas y columnas reales, pero la operación de agregación no corresponde a lo que pidió el usuario. No lo detecta un parser ni un EXPLAIN — hace falta comparar contra la intención estructurada (operation="average" en el optimizer).
