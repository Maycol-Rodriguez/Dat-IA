Validación determinística: sqlglot y EXPLAIN

Antes de gastar una llamada a un LLM juez, hay dos capas casi gratis que capturan los niveles 1 y 2 de error de la Clase 1: parsear el SQL sin ejecutarlo, y pedirle a Postgres un plan de ejecución sin correr la query de verdad.

1. El problema

Supón que Gemini alucina una columna que no existe en el esquema recuperado:

SQL con columna inexistente
SELECT carrier_name, on_time_percentage
FROM carriers
ORDER BY on_time_percentage DESC LIMIT 1;

El DDL real define on_time_rate, no on_time_percentage. Hoy ese SQL llega directo a execute_sql (app/main.py:991), Postgres lanza UndefinedColumn, y el usuario recibe el mensaje crudo: "La consulta generada falló al ejecutarse: ..." (app/main.py:1454-1459). No hay reintento ni segunda oportunidad.

2. El concepto desde cero

¿Qué es un AST y por qué parsear sin ejecutar?

Un AST (árbol de sintaxis abstracta) es la representación estructurada de una sentencia SQL: qué es un SELECT, cuáles son sus columnas, de qué tabla viene, qué condiciones tiene el WHERE. Un parser construye ese árbol sin conectarse a ninguna base de datos — es texto convertido en estructura, nada más.

sqlglot es una librería Python que parsea SQL de múltiples dialectos (incluido Postgres) a un AST navegable, sin dependencias de red ni de una BD real.

Parsear y detectar sintaxis inválida
import sqlglot

try:
    tree = sqlglot.parse_one(sql, read="postgres")
except sqlglot.errors.ParseError as exc:
    return SqlValidation(is_valid=False, stage="syntax", error=str(exc))

Extraer tablas del AST y compararlas contra el esquema recuperado

Una vez parseado, tree.find_all(sqlglot.exp.Table) devuelve cada tabla citada en la query. Dat-IA ya sabe qué tablas son "válidas para esta pregunta": son las que retrieve_ddl_context (app/main.py:615) recuperó — la combinación de suggested_tables del optimizer (app/optimizer/query_optimizer.py:64) y la búsqueda semántica en ChromaDB. Si el SQL cita una tabla fuera de ese conjunto, es alucinación de esquema, no un error de negocio — y se detecta sin tocar Postgres.

Verificar tablas contra el esquema recuperado
cited_tables = {t.name for t in tree.find_all(sqlglot.exp.Table)}
unknown = cited_tables - set(allowed_tables)

if unknown:
    return SqlValidation(
        is_valid=False,
        stage="tables",
        error=f"Tablas no reconocidas: {unknown}",
    )

Dry-run con EXPLAIN: validar columnas y tipos sin ejecutar

sqlglot no conoce el esquema completo de la base de datos (columnas, tipos, índices) — eso exigiría replicarlo dentro de la app y mantenerlo sincronizado. Postgres ya lo tiene. La forma barata de aprovecharlo sin correr la query real es pedirle un EXPLAIN (nunca EXPLAIN ANALYZE, que sí ejecuta): Postgres construye el plan de ejecución, lo que exige resolver cada tabla, columna y tipo mencionado, y lo hace sin leer una sola fila.

Dry-run con EXPLAIN (usa el sql_database ya existente en app/main.py)
def dry_run_explain(db: SQLDatabase, sql: str) -> str | None:
    """Devuelve None si EXPLAIN pasa, o el mensaje de error de Postgres."""
    result = db.run_no_throw(f"EXPLAIN {sql}", fetch="cursor")
    return result if isinstance(result, str) else None
⚠️

db.run_no_throw es el mismo patrón que ya usa execute_sql (app/main.py:1007): devuelve un str con el error en vez de lanzar una excepción. Reutilizar esa convención evita introducir un segundo estilo de manejo de errores en el mismo archivo.

Degradación cuando no hay base de datos

DATABASE_URL está vacío en .env.example, y sql_database puede quedar en None si la conexión a Supabase falla al arrancar (app/main.py:167-175). El dry-run debe degradarse solo — saltar el paso "dry_run" y quedarse con la validación de sintaxis + tablas — nunca convertir la ausencia de BD en un fallo de validación.

Endurecer lo que ya existe: mover las guardas de execute_sql

execute_sql (app/main.py:991-1005) ya bloquea sentencias que no son SELECT y el apilado de varias sentencias en una sola consulta:

app/main.py — guardas actuales dentro de execute_sql
def execute_sql(db: SQLDatabase, sql: str, row_limit: int = 200) -> dict:
    stripped = sql.strip().rstrip(";")

    if not re.match(r"(?is)^select\b", stripped):
        return {"error": "Solo se permiten sentencias SELECT."}

    if ";" in stripped:
        return {"error": "Solo se permite una sentencia SQL por consulta."}

    result = db.run_no_throw(stripped, fetch="cursor")
    ...

El problema no es que falten estas guardas — es dónde se aplican. Con el juez LLM (Clase 3) y el bucle de reintento (Clase 4) delante de execute_sql, un SQL que nunca fue un SELECT (o que apila dos sentencias) pasaría primero por una llamada completa al juez, y potencialmente por 1–2 regeneraciones, antes de que esta regex lo rechace al final. Es gastar presupuesto de LLM en algo que una regla de microsegundos ya sabía que iba a fallar. Por eso se mueven al validador: es el primer filtro, antes de gastar nada caro. También queda como el único punto de aplicación: si mañana otro camino de ejecución (por ejemplo el nodo execute del grafo de la Clase 6) llamara a db.run_no_throw directamente en vez de pasar por execute_sql, perdería esta protección; si vive en validate_sql — paso obligatorio antes de cualquier ejecución — se aplica una sola vez para cualquier camino.

Qué cambia al moverlas, no es solo copiar y pegar:

  • Solo-SELECT: hoy es una regex sobre el string crudo. En el validador ya tenemos el AST parseado (para sacar las tablas), así que conviene usar isinstance(tree, exp.Select) reutilizando ese mismo árbol — gratis, y más robusto que una regex.
  • Anti-stacking (; en el string): aquí hay un hallazgo que conviene conocer porque contradice la intuición — sqlglot no lo detecta solo. sqlglot.parse_one("SELECT 1; SELECT 2;") no lanza error: lo envuelve silenciosamente en un nodo exp.Block con ambas sentencias adentro (sqlglot es permisivo por diseño, sabe parsear scripts completos). Por eso la guarda explícita de ; sigue siendo necesaria, y debe correr antes de invocar sqlglot.parse_one, no después.
⚠️

La guarda de ; es un chequeo de substring, no de AST — un SQL legítimo con un literal de texto que contenga ; (WHERE description = 'a;b') daría falso positivo. Es una limitación que ya existe hoy en execute_sql sin cambios; se traslada, no se corrige. Corregirla de raíz exigiría contar sentencias reales en el AST, más trabajo del que esta guarda amerita hoy.

La guarda nueva: exigir LIMIT

Esta no es una guarda de seguridad — esa ya la cubre solo-SELECT. Es una guarda de costo de recursos, motivada por algo concreto que ya pasa en producción:

app/main.py:1012-1014 — el corte a row_limit llega demasiado tarde
rows = [dict(row) for row in result.mappings()]
return {"rows": rows[:row_limit]}

result.mappings() iterado dentro del list comprehension materializa el cursor completo en memoria de Python — y solo después [:row_limit] corta a 200. Si el SQL generado no trae LIMIT y la tabla tiene 100 mil filas sin filtro selectivo, Postgres calcula y transmite las 100 mil filas completas, Python construye 100 mil diccionarios, y recién ahí se descartan casi todos.

Con el AST ya parseado, detectar la ausencia (o el exceso) de LIMIT es gratis:

Leer y ajustar el LIMIT sobre el AST
tree.find(exp.Limit)  # None si no hay LIMIT

tree.limit(200)  # añade LIMIT 200 si no existía
tree.args["limit"] = exp.Limit(expression=exp.Literal.number(200))  # capar uno mayor a 200

Hay dos formas de resolverlo: (A) rechazar el SQL sin LIMIT y devolverlo al bucle de reintento de la Clase 4, o (B) auto-inyectar o capar el LIMIT directamente en el AST y reserializar con tree.sql(dialect="postgres"), sin gastar ningún intento de reintento. La opción B es la recomendada: no es una discrepancia semántica que el juez deba opinar, es un ajuste mecánico y determinístico — gastar un intento de LLM en esto desperdiciaría el presupuesto del bucle de reintento en algo que se resuelve con una línea de sqlglot. El valor máximo se alinea con el mismo row_limit=200 que ya usa execute_sql, para que el límite se aplique una sola vez, con el mismo valor, en toda la cadena.

3. Cómo se integra en Dat-IA

Nuevo módulo app/validation/sql_validator.py, con una única función pública que compone las capas en orden barato → caro:

app/validation/sql_validator.py — firma propuesta
from dataclasses import dataclass
from typing import Literal

@dataclass(frozen=True)
class SqlValidation:
    is_valid: bool
    stage: Literal["syntax", "statement", "tables", "dry_run", "ok"]
    error: str

def validate_sql(
    sql: str,
    allowed_tables: list[str],
    db: SQLDatabase | None = None,
) -> SqlValidation:
    # 1. anti-stacking ";" en el string — antes de parsear, sqlglot no lo detecta solo
    # 2. sqlglot.parse_one → captura errores de sintaxis
    # 3. isinstance(tree, exp.Select) — reemplaza la regex de solo-SELECT
    # 4. comparar tablas citadas (excluyendo alias de CTE) contra allowed_tables
    # 5. exigir/capar LIMIT sobre el AST (recomendado: auto-inyectar, no rechazar)
    # 6. si db is not None: EXPLAIN como dry-run
    ...

4. Errores comunes

  • Mantener un esquema paralelo en Python. No repliques columnas y tipos a mano para "validar sin BD" — es exactamente el tipo de estado duplicado que se desincroniza. Para eso está el dry-run con EXPLAIN.
  • Usar EXPLAIN ANALYZE por error. Esa variante sí ejecuta la query. El dry-run debe ser EXPLAIN a secas.
  • Romper el flujo cuando no hay BD. El validador debe reportar stage="dry_run" como no aplicable, no como fallo, cuando db is None — igual que /ready ya reporta "not_configured" (app/main.py:1084-1094) sin romper el resto de la API.
  • Confundir "tabla no reconocida" con "tabla inexistente en la BD real". Aquí se valida contra el conjunto que Dat-IA decidió que era relevante para esta pregunta, no contra el catálogo completo de la base de datos — una tabla real pero fuera de contexto también debe rechazarse, porque señala que el LLM se salió del DDL que se le dio.

5. Checklist de implementación

  • Añadir sqlglot a pyproject.toml (uv add sqlglot).
  • Crear app/validation/sql_validator.py con SqlValidation y validate_sql(...).
  • Mover las guardas de solo-SELECT (ahora vía isinstance(tree, exp.Select)) y anti-stacking (chequeo de ;, antes de parsear) desde execute_sql hacia el validador.
  • Añadir la guarda de LIMIT: auto-inyectar tree.limit(200) si falta, o capar el existente si excede 200.
  • Implementar dry_run_explain reutilizando db.run_no_throw, con manejo explícito de db is None.
  • Tests: SQL sintácticamente inválido, SQL con tabla fuera de allowed_tables, SQL con columna inexistente (vía dry-run con SQLite o mock), y el caso db=None.

Comprueba que entendiste

Porque sqlglot solo conoce la estructura sintáctica del SQL (el AST), no el esquema real de la base de datos. No sabe qué columnas existen en carriers a menos que se le enseñe ese esquema aparte. Por eso se delega esa verificación a Postgres mediante EXPLAIN, que sí conoce el esquema real.
EXPLAIN construye el plan de ejecución sin leer datos reales — valida sintaxis, tablas, columnas y tipos. EXPLAIN ANALYZE además ejecuta la query de verdad para medir tiempos reales. Para un dry-run se usa siempre EXPLAIN a secas.
Debe completar las etapas de sintaxis y tablas normalmente, y omitir la etapa de dry-run sin marcarla como error — igual que /ready reporta "no configurada" sin romper el resto de la API. El validador nunca debe fallar porque falte infraestructura opcional.
Porque con el juez LLM y el bucle de reintento delante de execute_sql, un SQL que nunca fue un SELECT pasaría primero por una llamada completa al juez y posibles regeneraciones antes de ser rechazado por una regla trivial al final — desperdiciando presupuesto de LLM en algo que un chequeo de microsegundos ya sabía que iba a fallar. Moverlas al validador las convierte además en el único punto de aplicación para cualquier camino de ejecución futuro.
Porque sqlglot es permisivo por diseño: en vez de lanzar un error, envuelve silenciosamente varias sentencias en un nodo exp.Block. La guarda explícita de ; sigue siendo necesaria y debe correr antes de invocar el parser, no depender de que este falle.
Porque la ausencia de LIMIT no es una discrepancia semántica que el juez deba evaluar — es un ajuste mecánico y determinístico. Gastar un intento del bucle de reintento (que ya tiene un máximo de 2) en algo que se resuelve con una línea de sqlglot desperdicia ese presupuesto en un caso sin ambigüedad de negocio.
