El bucle de auto-corrección (evaluator-optimizer)

Con el validador determinístico (Clase 2) y el juez LLM (Clase 3) ya se puede detectar que un SQL está mal. Falta la pieza que los conecta: qué hacer cuando algo falla. Esta clase cierra el ciclo generador → crítico → regenerar, sin frameworks nuevos.

1. El problema

Hoy, si build_rag_response produce un SQL malo, no hay segunda oportunidad: se ejecuta tal cual, y si Postgres lo rechaza, el usuario recibe el error crudo (app/main.py:1330-1336). Con un validador y un juez, aparece una tercera opción entre "ejecutar" y "fallar": volver a pedirle al generador que lo intente de nuevo, mostrándole exactamente qué estuvo mal.

2. El concepto desde cero: evaluator-optimizer

Es uno de los patrones agénticos más simples y rentables: un generador produce una salida, un evaluador la juzga, y si falla, el generador vuelve a intentarlo con el feedback del evaluador incluido en el prompt. No es "pedir lo mismo otra vez" — es pedir lo mismo con la información de qué falló la vez anterior, lo cual cambia mucho el resultado.

Por qué el límite de intentos es obligatorio, no opcional

Cada intento cuesta latencia (una llamada más al generador, otra al juez) y dinero (más tokens). Y hay un límite práctico de utilidad: si el generador falló dos veces con feedback específico de qué corregir, la probabilidad de que un tercer intento acierte cae mucho — normalmente el problema es que el DDL recuperado no tiene la información necesaria, no que el generador necesite "una oportunidad más". Por eso max_attempts = 2 es la política de Dat-IA: falla rápido y honesto en vez de insistir indefinidamente.

3. Cómo se implementa (sin framework)

Es un for con un contador, dentro de una nueva función en app/main.py que envuelve el ciclo generar → validar → juzgar:

app/main.py — generate_validated_sql (nuevo)
def generate_validated_sql(
    question: str,
    ddl: str,
    optimized_query: OptimizedQuery,
    allowed_tables: list[str],
    memory_examples: list[dict] | None = None,
    max_attempts: int = 2,
) -> tuple[RAGResponse, SqlVerdict | None, int]:
    """Genera SQL con hasta max_attempts intentos, validando y juzgando cada uno."""
    feedback: SqlVerdict | None = None

    for attempt in range(1, max_attempts + 1):
        rag_response = build_rag_response(question, ddl, memory_examples, feedback=feedback)

        validation = validate_sql(rag_response.sql, allowed_tables, db=sql_database)
        if not validation.is_valid:
            feedback = SqlVerdict(
                is_valid=False, answers_question=False,
                issues=[validation.error], suggested_fix="", confidence=0.0,
            )
            continue

        verdict = judge_sql(optimized_query, rag_response.sql, ddl)
        if verdict.is_valid and verdict.answers_question:
            return rag_response, verdict, attempt

        feedback = verdict  # se usa en el siguiente intento

    return rag_response, feedback, max_attempts  # se agotaron los intentos

Cómo entra el feedback al prompt

build_rag_response (app/main.py:889) recibe un parámetro nuevo opcional feedback: SqlVerdict | None. Cuando no es None, se añade una sección extra al prompt existente, en el mismo estilo que ya usa la sección de memoria (app/main.py:919-921):

Sección nueva del prompt en build_rag_response
### Previous attempt feedback
Your previous SQL was rejected for these reasons:
{feedback.issues}
Suggested fix: {feedback.suggested_fix}
Do not repeat the same mistake.

Salida honesta cuando no se logra

Si se agotan los intentos, /query/answer no debe ejecutar el último SQL "por si acaso" — debe responder con transparencia qué falló, dejando el SQL candidato visible para auditoría (no para que el usuario lo corra). Esto exige ampliar AnswerResponse (app/main.py:277):

app/main.py — campos nuevos en AnswerResponse
class AnswerResponse(BaseModel):
    answer: str
    sql: str
    data: list[dict]
    sources: str
    status: str
    table: ResultTable | None = None
    attempts: int = 1
    validation: str | None = None
    warnings: list[str] = []

Efecto sobre la memoria: endurecer validated=True

Esta es la corrección directa del hueco #2 de la portada. Hoy (app/main.py:1370-1377) validated=True se otorga con solo tener execution_status="success". A partir de esta clase, ese flag exige las tres condiciones: juez aprobado (verdict.is_valid and verdict.answers_question), ejecución sin error, y guardrail de resultado aprobado (Clase 5). Un SQL que ejecuta pero el juez rechazó nunca debe reforzarse como ejemplo futuro.

4. Errores comunes

  • Reintentar sin cambiar el prompt. Si el segundo intento recibe exactamente el mismo prompt que el primero, es casi seguro que se repite el mismo error — el feedback del juez tiene que llegar al generador.
  • No poner límite, o ponerlo muy alto. Cada intento fallido ya reveló que el problema probablemente no es "mala suerte" sino algo estructural (DDL insuficiente, pregunta ambigua). Insistir más allá de 2 intentos rara vez ayuda y siempre cuesta.
  • Ejecutar el SQL del último intento aunque el juez lo haya rechazado. Es preferible responder "no pude" que ejecutar algo que ya se sabe que está mal.

5. Checklist de implementación

  • Añadir parámetro feedback: SqlVerdict | None a build_rag_response y la sección de prompt correspondiente.
  • Implementar generate_validated_sql(...) en app/main.py con el bucle for y max_attempts=2.
  • Ampliar AnswerResponse con attempts, validation, warnings.
  • Endurecer la condición de validated=True en /query/answer (app/main.py:1370-1377) para exigir juez + ejecución + guardrail.
  • Tests: caso que falla en el intento 1 y se corrige en el 2; caso que agota los 2 intentos y responde "no pude" sin ejecutar nada.

Comprueba que entendiste

Reintentar sin feedback repite el mismo prompt y muy probablemente produce el mismo error. Reintentar con feedback añade al prompt exactamente qué falló en el intento anterior (issues del juez o error del validador), dando al generador información nueva para corregir, no solo una segunda tirada de dados.
Cada intento añade latencia y costo de API. Si dos intentos con feedback específico no bastan, el problema suele ser estructural (esquema insuficiente, pregunta ambigua) y no algo que un tercer o cuarto intento vaya a resolver. Es más barato y más honesto fallar rápido.
Porque el bucle es lo que produce un veredicto del juez además del resultado de ejecución. Antes, "validated" solo dependía de que la ejecución no lanzara error; ahora puede exigir también que el juez haya aprobado, cerrando la vía por la que hoy se contamina la memoria V2 con SQL semánticamente incorrecto.
