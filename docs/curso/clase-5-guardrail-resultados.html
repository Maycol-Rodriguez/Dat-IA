Guardrail de resultados y groundedness

El SQL ya pasó el validador determinístico y el juez (Clases 2–3). Se ejecutó sin error. Queda un último punto ciego: el LLM que redacta la respuesta en español a partir de las filas puede inventar o redondear números que no están ahí.

1. El problema

El pregunta original del usuario decidió que este evaluador fuera un guardrail ligero, no un segundo juez semántico completo — esa cobertura ya la da el juez de la Clase 3 antes de ejecutar. Lo que queda sin cubrir es otra cosa: synthesize_answer (app/main.py:987-1005) recibe las filas reales y redacta texto libre en español. Un LLM redactando texto libre puede, por ejemplo, decir "el ingreso promedio fue de S/ 452" cuando las filas devueltas dicen 438.

💡

Por qué no un segundo juez semántico completo: duplicaría casi todo el trabajo de la Clase 3 (que ya verificó operación, métricas, filtros) y añadiría otra llamada a LLM completa antes de responder. El guardrail de esta clase es deliberadamente más barato: chequeos determinísticos sobre las filas, más una verificación de groundedness acotada al texto final.

2. El concepto desde cero

Chequeos determinísticos sobre el resultado (sin LLM)

Antes de tocar el texto redactado, hay señales de alerta que se detectan mirando solo las filas — gratis, sin llamar a ningún modelo:

  • Resultado vacío (len(rows) == 0): la pregunta era válida pero el filtro no encontró nada; merece un mensaje distinto a un error.
  • Todas las columnas de métrica en NULL: el JOIN pudo ser correcto pero la columna que se quería agregar vino vacía en todas las filas.
  • Resultado truncado silenciosamente: execute_sql (app/main.py:937-960) corta a row_limit=200 sin avisar. Si len(rows) == row_limit, es señal de que probablemente hay más filas de las mostradas — hoy eso no se comunica al usuario.
  • Cardinalidad inesperada según la operación: si optimized_query.operation es "rank_desc" con LIMIT 1 implícito en la pregunta pero el SQL devuelve 40 filas, algo no cuadra con la intención.
app/validation/result_guardrail.py — chequeo de resultado
from dataclasses import dataclass

@dataclass(frozen=True)
class ResultCheck:
    ok: bool
    warnings: list[str]

def check_result(
    rows: list[dict],
    optimized_query: OptimizedQuery,
    row_limit: int = 200,
) -> ResultCheck:
    warnings = []

    if not rows:
        warnings.append("La consulta no devolvió filas.")

    if len(rows) == row_limit:
        warnings.append(
            f"El resultado se truncó a {row_limit} filas; puede haber más datos."
        )

    for metric in optimized_query.metrics:
        if rows and all(row.get(metric) is None for row in rows):
            warnings.append(f"La métrica '{metric}' vino vacía en todas las filas.")

    return ResultCheck(ok=not warnings, warnings=warnings)

Groundedness: que cada número en la respuesta exista en las filas

"Groundedness" significa que una afirmación está respaldada por la evidencia disponible — aquí, que cada cifra que aparece en el texto redactado por synthesize_answer aparezca (con tolerancia de redondeo) entre los valores de rows. Es un chequeo barato: extraer números del texto con una expresión regular y comparar contra los valores numéricos de las filas, sin necesidad de otro LLM para la primera pasada.

app/validation/result_guardrail.py — groundedness
import re

@dataclass(frozen=True)
class GroundednessCheck:
    ok: bool
    unsupported_numbers: list[str]

def check_groundedness(
    answer: str,
    rows: list[dict],
    tolerance: float = 0.01,
) -> GroundednessCheck:
    numbers_in_answer = re.findall(r"\d+[.,]?\d*", answer)
    row_values = {
        round(float(value), 2)
        for row in rows for value in row.values()
        if isinstance(value, (int, float))
    }

    unsupported = []
    for raw in numbers_in_answer:
        candidate = float(raw.replace(",", ""))
        if not any(abs(candidate - v) <= tolerance * max(abs(v), 1) for v in row_values):
            unsupported.append(raw)

    return GroundednessCheck(ok=not unsupported, unsupported_numbers=unsupported)
⚠️

Este chequeo tiene falsos positivos esperables: números de fila (#1), conteos derivados (len(rows), que synthesize_answer ya menciona explícitamente según su prompt, app/main.py:995) o porcentajes calculados a partir de dos valores de la fila. La mitigación no es afinar la regex infinitamente, sino usarlo como señal de sospecha, no como bloqueo automático: si hay números sin respaldo, se regenera la redacción una vez con una instrucción más estricta; si persiste, se entrega igual pero con warnings visibles.

3. Cómo se integra en Dat-IA

Ambos chequeos se insertan en /query/answer (app/main.py:1264-1387) entre execute_sql y el guardado en memoria V2:

app/main.py — /query/answer con guardrail (fragmento nuevo)
execution = execute_sql(sql_database, rag_response.sql)
if "error" in execution:
    ...  # sin cambios

rows = execution["rows"]
result_check = check_result(rows, optimized_query)

answer_text = synthesize_answer(answer_llm, request.question, rag_response.sql, rows)
groundedness = check_groundedness(answer_text, rows)

if not groundedness.ok:
    # una sola regeneración con instrucción estricta, no un bucle nuevo
    answer_text = synthesize_answer(
        answer_llm, request.question, rag_response.sql, rows,
        strict_numbers=True,
    )
    groundedness = check_groundedness(answer_text, rows)

is_fully_validated = (
    verdict.is_valid and verdict.answers_question
    and result_check.ok and groundedness.ok
)
# is_fully_validated reemplaza el validated=True incondicional de app/main.py:1370-1377

4. Errores comunes

  • Convertir esto en un segundo juez semántico. Si empieza a preguntarse "¿esta respuesta tiene sentido de negocio?", ya se está duplicando la Clase 3. El guardrail se queda en resultado + groundedness, nada más.
  • Bloquear en vez de regenerar una vez. Un solo reintento de redacción con instrucción estricta resuelve la mayoría de los casos sin añadir demasiada latencia; más de un reintento aquí ya no es proporcional al problema.
  • Ignorar la truncación silenciosa. row_limit=200 ya existe en execute_sql pero hoy nadie lo comunica al usuario — es un warning gratis de añadir.

5. Checklist de implementación

  • Crear app/validation/result_guardrail.py con ResultCheck, check_result, GroundednessCheck, check_groundedness.
  • Añadir parámetro strict_numbers: bool = False a synthesize_answer (app/main.py:987) que endurece la instrucción del prompt cuando es True.
  • Insertar ambos chequeos en /query/answer entre ejecución y guardado de memoria; propagar warnings en AnswerResponse (ya ampliado en la Clase 4).
  • Tests: filas vacías, métrica en NULL, resultado truncado a row_limit, respuesta con número inventado que dispara el reintento de redacción.

Comprueba que entendiste

Porque un segundo juez semántico duplicaría gran parte del trabajo que ya hace el juez de la Clase 3 antes de ejecutar, añadiendo latencia y costo sin cubrir un fallo nuevo. El fallo real que queda descubierto tras la ejecución es otro: la redacción en lenguaje natural puede inventar cifras — eso es lo que ataca el guardrail.
Porque comparar números extraídos del texto contra los valores de las filas es una operación determinística y barata — no requiere juicio ni comprensión de lenguaje, solo coincidencia numérica con tolerancia de redondeo. Reservar un LLM para esto sería gastar una llamada cara en un chequeo mecánico.
Hoy la truncación es silenciosa: el usuario recibe 200 filas sin saber que podría haber más. Con check_result, si len(rows) == row_limit se añade un warning explícito en la respuesta, avisando que el resultado puede estar incompleto.
