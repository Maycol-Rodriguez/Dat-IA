De pipeline a agente conversacional con LangGraph

Las Clases 2 a 5 le dieron al pipeline un ciclo de validación y reintento — pero sigue siendo stateless: cada request de /query/answer no sabe nada de la anterior. Esta clase envuelve ese mismo pipeline (sin reescribirlo) en un grafo con estado para que la interacción se sienta como una conversación.

1. El problema

Un usuario pregunta "¿Cuál fue el ingreso total por mes en 2018?", recibe la respuesta, y luego pregunta "¿y en 2017?". Hoy esa segunda pregunta llega sola a optimize_query (app/optimizer/query_optimizer.py:83): no hay "ingreso total por mes" en ningún lado, solo "2017". El optimizer no tiene forma de saber que el usuario quiere repetir la misma métrica con otro año — cada request es una isla.

2. El concepto desde cero: qué es un grafo de estados

LangGraph es una librería para construir flujos como un grafo explícito en vez de una secuencia de llamadas a función. Tres piezas:

  • Nodo: una función que recibe el estado actual y devuelve un delta (los campos que cambia). No devuelve el estado completo, solo lo nuevo.
  • Arista: la conexión que dice "después de este nodo, ve a este otro".
  • Arista condicional: una función que, dado el estado, decide a cuál de varios nodos ir. Es lo que hace posible ciclos (volver a generar SQL) y ramas (clarificar en vez de responder).

La diferencia con el pipeline actual no es filosófica, es práctica: hoy /query/answer es una secuencia fija de llamadas a función en Python. Con LangGraph, la misma secuencia se declara como grafo, lo que permite dos cosas que un for lineal no da gratis: persistencia de estado entre llamadas (necesaria para conversación) y visualización/trazabilidad de cada paso.

El estado: qué viaja entre nodos

Se define como un TypedDict. messages usa el reducer add_messages, que en vez de sobrescribir la lista, anexa los mensajes nuevos — así el historial de la conversación se acumula automáticamente turno a turno.

app/agent/graph.py — estado del grafo
from typing import Annotated, TypedDict
from langgraph.graph.message import add_messages

class DatIAState(TypedDict):
    messages: Annotated[list, add_messages]
    question: str
    resolved_question: str
    optimized: OptimizedQuery | None
    ddl: str
    allowed_tables: list[str]
    sql: str
    verdict: SqlVerdict | None
    rows: list[dict]
    answer: str
    attempts: int
    needs_clarification: bool

Los nodos: el pipeline existente, envuelto, no reescrito

Cada nodo es una función delgada que llama al código que ya existe y lo adapta a la forma "recibe estado, devuelve delta". No se reimplementa ninguna lógica de negocio:

app/agent/graph.py — nodos como envoltorios finos
def optimize_node(state: DatIAState) -> dict:
    optimized = optimize_query(state["resolved_question"], llm=optimizer_llm)
    return {"optimized": optimized}

def retrieve_ddl_node(state: DatIAState) -> dict:
    resp = retrieve_ddl_context(
        text_collection,
        state["optimized"].normalized_question,
        suggested_tables=state["optimized"].suggested_tables,
    )
    return {"ddl": resp.ddl, "allowed_tables": resp.tabla}

def generate_sql_node(state: DatIAState) -> dict:
    rag_response, verdict, attempts = generate_validated_sql(
        state["resolved_question"], state["ddl"], state["optimized"], state["allowed_tables"],
    )
    return {"sql": rag_response.sql, "verdict": verdict, "attempts": attempts}

# execute_node envuelve execute_sql, synthesize_node envuelve synthesize_answer,
# guardrail_node envuelve check_result/check_groundedness (Clase 5), etc.

El resto de nodos (shield_node, memory_lookup_node, execute_node, guardrail_node, synthesize_node, save_memory_node) siguen el mismo patrón: envuelven classify_shield, _search_query_memory_v2_examples, execute_sql, los chequeos de la Clase 5, synthesize_answer y upsert_query_memory_v2 respectivamente, sin cambiar su lógica interna.

Aristas condicionales: dónde aparecen los ciclos y las ramas

Son funciones que devuelven el nombre del siguiente nodo según el estado. El bucle de la Clase 4 (generar → validar → juzgar → regenerar) ya vive dentro de generate_validated_sql como un for interno — no hace falta un ciclo de grafo para eso. Donde el ciclo/rama de LangGraph sí aporta es en decisiones de alto nivel: si la pregunta necesita aclaración, si ejecutar falló y hay que regenerar.

app/agent/graph.py — arista condicional de ruteo
def route_after_optimize(state: DatIAState) -> str:
    optimized = state["optimized"]
    if not optimized.metrics and not optimized.date_range:
        return "clarify"
    return "retrieve_ddl"

graph.add_conditional_edges("optimize", route_after_optimize, {
    "clarify": "clarify",
    "retrieve_ddl": "retrieve_ddl",
})

Checkpointer y thread_id: qué persiste y dónde

El checkpointer es lo que guarda el DatIAState completo después de cada nodo, indexado por un thread_id — el identificador de una conversación. En desarrollo, MemorySaver guarda todo en memoria de proceso (se pierde al reiniciar). En producción, PostgresSaver persiste contra la misma Supabase que ya usa create_db_engine (app/db/connect_db.py), así la conversación sobrevive a un reinicio del contenedor.

app/agent/graph.py — compilar el grafo con checkpointer
from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph import StateGraph, START, END

graph = StateGraph(DatIAState)
graph.add_node("resolve_question", resolve_question_node)
graph.add_node("shield", shield_node)
graph.add_node("optimize", optimize_node)
graph.add_node("clarify", clarify_node)
graph.add_node("retrieve_ddl", retrieve_ddl_node)
graph.add_node("generate_sql", generate_sql_node)
graph.add_node("execute", execute_node)
graph.add_node("guardrail", guardrail_node)
graph.add_node("synthesize", synthesize_node)
graph.add_node("save_memory", save_memory_node)

graph.add_edge(START, "resolve_question")
graph.add_edge("resolve_question", "shield")
graph.add_edge("shield", "optimize")
# ... conditional edges de route_after_optimize y route_after_execute ...
graph.add_edge("save_memory", END)

# En el lifespan de app/main.py, junto a los demás clientes globales:
checkpointer = MemorySaver()
compiled_graph = graph.compile(checkpointer=checkpointer)

Reescritura de referencias: el nodo que hace que se sienta conversacional

"¿Y en 2018?" solo tiene sentido junto al mensaje anterior. El nodo resolve_question corre antes del optimizer: toma state["messages"] (el historial que persiste el checkpointer) y la pregunta cruda, y le pide a un LLM que produzca una pregunta autónoma y completa.

app/agent/graph.py — resolve_question_node
def resolve_question_node(state: DatIAState) -> dict:
    if len(state["messages"]) <= 1:
        return {"resolved_question": state["question"]}  # primer turno, nada que resolver

    prompt = f"""
Dado este historial de conversación y la nueva pregunta del usuario,
reescribe la nueva pregunta para que sea autónoma y completa, sin
depender del contexto previo.

Historial: {state['messages'][-4:]}
Nueva pregunta: {state['question']}
"""
    resolved = optimizer_llm.invoke(prompt).content
    return {"resolved_question": resolved}

Clarificación: cuándo preguntar en vez de adivinar

El nodo clarify corta el pipeline antes de gastar una llamada de generación de SQL cuando la pregunta optimizada no trae ni métrica ni filtro reconocible — señal de que adivinar produciría una respuesta plausible pero probablemente incorrecta. Es mejor interrumpir con "¿te refieres a ingresos, pedidos o algo más?" que forzar una respuesta.

3. El endpoint nuevo: /chat

Convive con /query/answer, que se queda como API stateless (útil para integraciones que no necesitan memoria conversacional). /chat recibe thread_id explícito:

app/main.py — endpoint /chat (nuevo)
class ChatRequest(BaseModel):
    question: str = Field(..., min_length=1)
    thread_id: str

@app.post("/chat")
async def chat(request: ChatRequest):
    config = {"configurable": {"thread_id": request.thread_id}}
    result = await compiled_graph.ainvoke(
        {"question": request.question, "messages": [{"role": "user", "content": request.question}]},
        config=config,
    )
    return {"answer": result["answer"], "sql": result["sql"], "needs_clarification": result["needs_clarification"]}

El grafo se compila una vez en el lifespan actual (app/main.py:73), junto a los demás clientes globales — mismo patrón que ya usa la app para rag_llm, chroma_client, etc. Para streaming de la respuesta en tiempo real, LangGraph expone .astream(); un /chat/stream con Server-Sent Events queda como extensión natural (Clase 7).

4. Errores comunes

  • Reescribir la lógica de negocio dentro de los nodos. Un nodo que duplica lo que ya hace build_rag_response en vez de llamarlo introduce dos fuentes de verdad. Los nodos deben ser delgados.
  • Olvidar el reducer add_messages. Sin él, cada nodo que toque messages sobrescribe el historial completo en vez de anexar — se pierde la conversación turno a turno.
  • MemorySaver en producción. Es solo para desarrollo: vive en el proceso, se pierde al reiniciar el contenedor. Producción necesita PostgresSaver u otro backend persistente.
  • No poner límite a los ciclos del grafo. Igual que el bucle manual de la Clase 4 necesita max_attempts, cualquier ciclo a nivel de grafo (por ejemplo, reintentar ejecución tras un fallo) necesita un contador en el estado para no entrar en un ciclo infinito.

Comprueba que entendiste

El nodo resolve_question, que corre antes del optimizer y usa el historial persistido por el checkpointer (state["messages"]) para reescribir la pregunta corta en una pregunta autónoma y completa antes de que optimize_query la procese.
Porque guarda el estado en memoria del proceso Python — se pierde por completo si el contenedor se reinicia o se despliega una nueva versión. Producción necesita un checkpointer persistente como PostgresSaver, respaldado por una base de datos real.
Para no duplicar la lógica de negocio en dos lugares. Los nodos son envoltorios delgados que adaptan las funciones ya probadas del pipeline a la forma "recibe estado, devuelve delta" que exige LangGraph — el comportamiento de negocio no cambia, solo la orquestación alrededor.
