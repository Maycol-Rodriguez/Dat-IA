Qué más añadir: roadmap priorizado

Con el validador, el juez, el bucle de reintento, el guardrail y el agente conversacional (Clases 2–6), el workflow objetivo del diagrama de arquitectura del equipo queda cubierto en su mayor parte. Esto es lo siguiente, ordenado por relación esfuerzo/impacto.

Alto impacto, bajo esfuerzo

Feedback 👍 / 👎 del usuario sobre la memoria

La memoria V2 hoy decide validated solo con señales automáticas (juez + ejecución + guardrail, tras la Clase 4). Un simple endpoint POST /memory/v2/{'{memory_id}'}/feedback que ajuste validated según lo que el usuario de negocio confirma añade una señal humana que ningún LLM puede replicar: si la respuesta era o no lo que realmente necesitaba, independientemente de si el SQL "se veía" correcto.

Explicación del SQL ejecutado (auditoría)

Un endpoint o campo que traduzca el SQL final a una explicación en español de qué tablas usó y qué filtros aplicó — reutilizando el mismo answer_llm ya inicializado en el lifespan — ayuda a que un analista de negocio confíe en la respuesta sin tener que leer SQL.

Sugerencias de follow-up

El OptimizedQuery estructurado (intent, metrics, group_by) ya contiene suficiente información para sugerir la siguiente pregunta lógica ("¿quieres verlo desglosado por mes?") sin ningún costo adicional de LLM — son reglas sobre campos que ya se calculan hoy.

Caché exacto por hash de pregunta

Antes de correr todo el pipeline (shield, optimizer, retrieval, generación), un hash de la pregunta normalizada contra un caché simple evita repetir el trabajo completo para preguntas idénticas repetidas en poco tiempo — distinto de la memoria semántica V2, que ya cubre preguntas similares pero no exactas.

Alto impacto, esfuerzo medio

LangSmith: trazas por nodo + dataset de evaluación offline

El diagrama de arquitectura del equipo ya contempla LangSmith en el bloque de "Observabilidad y monitoreo". Con LangGraph (Clase 6) cada nodo queda trazado automáticamente si se configura LANGCHAIN_TRACING_V2 y las variables de LangSmith — no requiere instrumentación manual adicional. Además, los benchmarks que ya existen en reports/ (query_memory_v2_operation_benchmark.json, query_memory_v2_threshold_calibration.json) son la base natural de un dataset de evaluación offline: casos de pregunta → SQL esperado que corren contra cada cambio de prompt antes de desplegarlo.

Agente de visualización

El bloque "Parseo y Evaluación" del README del proyecto menciona explícitamente "creación de gráficos automáticos". format_result_table (app/formatting/result_formatter.py) ya clasifica columnas por tipo (percentage, date, decimal, etc.) — esa clasificación, cruzada con optimized_query.operation (por ejemplo "temporal_trend" → línea, "rank_desc" → barras horizontales), es suficiente para elegir un tipo de gráfico sin necesitar otro LLM.

Streaming con Server-Sent Events

LangGraph expone .astream() de forma nativa sobre el grafo compilado en la Clase 6. Un endpoint /chat/stream que vaya emitiendo eventos por nodo (“validando SQL…”, “ejecutando…”) mejora la percepción de latencia sin cambiar la lógica interna.

Infraestructura

Presupuesto de tokens y timeout por nodo

Con más LLMs en el camino crítico (generador, optimizer, juez, síntesis, reescritura de preguntas), un timeout explícito por nodo del grafo evita que una sola llamada lenta cuelgue toda la conversación — hoy no existe ningún timeout configurado en las llamadas a Gemini dentro de app/main.py.

Métricas a Grafana

El mismo diagrama de arquitectura menciona Grafana junto a LangSmith. Métricas operativas (tasa de rechazo del juez, intentos promedio antes de éxito, latencia por nodo) son un complemento natural a las trazas de LangSmith — LangSmith explica por qué falló un caso puntual; Grafana muestra la tendencia agregada en el tiempo.

Comprueba que entendiste

Porque el juez LLM verifica que el SQL implemente correctamente la estructura detectada por el optimizer, pero no puede saber si esa estructura realmente capturó lo que el usuario de negocio necesitaba en un sentido más amplio (por ejemplo, si la definición de "ingreso" que usó coincide con la que maneja el área de finanzas). Esa validación solo la puede dar una persona.
Porque LangGraph traza automáticamente cada nodo del grafo cuando se configuran las variables de entorno de LangSmith — no hace falta instrumentar manualmente cada función con logging de trazas; el grafo ya expone su ejecución paso a paso de forma nativa.
✅

Con las Clases 1–6 implementadas, los tres huecos de la portada quedan cerrados: hay validación antes de ejecutar, la memoria solo se refuerza con SQL verificado, y la interacción es conversacional con estado persistente. Este roadmap es para después de esa base, no un requisito para completarla.
