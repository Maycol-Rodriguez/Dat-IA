Curso interno · Arquitectura Dat-IA

De pipeline Text-to-SQL a agente conversacional validado

Esta es la guía para la próxima actualización de arquitectura de Dat-IA: añadir un validador de SQL, un evaluador ligero de resultados y convertir el pipeline actual en un agente conversacional. Siete clases cortas, desde cero, con el código real del proyecto.

El problema, con código real

Hoy /query/answer genera SQL con Gemini y lo ejecuta directo. No hay ningún punto de control entre "el LLM escribió algo" y "eso se corre contra Supabase".

⚠️

3 huecos verificados en app/main.py:

  • Sin validación previa a la ejecución. El SQL de build_rag_response va directo a execute_sql. Si el SQL ejecuta pero responde otra pregunta (métrica equivocada, JOIN que duplica filas), pasa como éxito.
  • Memoria contaminada. /query/answer marca validated=True con solo haber ejecutado sin error. Ese SQL se reinyecta luego como ejemplo autoritativo para generar nuevo SQL.
  • Sin conversación. Cada request es independiente: no hay thread_id ni historial. "¿Y en 2018?" es imposible de responder.

Arquitectura objetivo

Fase 1 (manual, sin frameworks nuevos): un validador determinístico, un juez LLM y un bucle de reintento con límite. Fase 2 (LangGraph): el mismo pipeline envuelto en un grafo con estado, para que la interacción sea conversacional.

💡

Las cajas naranjas son nodos nuevos (validador, juez, guardrail, clarificación). Las cajas azules son estado/orquestación nueva. Las cajas grises son código que ya existe y solo se reutiliza.

Cómo usar este curso

Cada clase tiene la misma estructura: el problema con un caso de Olist, el concepto explicado desde cero, cómo se ve en el código de Dat-IA (con ruta de archivo), errores comunes, un checklist de implementación y unas preguntas para comprobar que quedó claro.
